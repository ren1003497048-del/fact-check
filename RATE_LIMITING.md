## 🎯 设计目标

1. **防止滥用**：阻止脚本、爬虫和恶意用户过度消耗 API 配额

## 📊 配置说明

### 三层速率限制架构

```typescript
RATE_LIMIT: {
  // 第一层：短期防护（防突发）
  SHORT_TERM: {
    WINDOW_MS: 60 * 1000,           // 1分钟
    MAX_REQUESTS: 5,                 // 最多5次
    MESSAGE: '请求过于频繁，请稍后再试...',
  },

  // 第二层：中期防护（防持续高频）
  MEDIUM_TERM: {
    WINDOW_MS: 15 * 60 * 1000,      // 15分钟
    MAX_REQUESTS: 15,                // 最多15次
    MESSAGE: '短时间内请求过多，请15分钟后再试...',
  },

  // 第三层：长期防护（防滥用）
  LONG_TERM: {
    WINDOW_MS: 60 * 60 * 1000,      // 1小时
    MAX_REQUESTS: 50,                // 最多50次
    MESSAGE: '请求次数超过限额，请1小时后再试...',
  },
}
```

### 限制对比表

| 层级 | 时间窗口 | 请求上限 | 平均速率 | 防护目标 |
|------|---------|---------|---------|---------|
| **短期** | 1分钟 | 5次 | 5次/分钟 | 🔒 防止突发请求 |
| **中期** | 15分钟 | 15次 | 1次/分钟 | 🛡️ 防止持续高频 |
| **长期** | 1小时 | 50次 | 0.83次/分钟 | 🚫 防止长期滥用 |

## 💡 设计理念

### 为什么使用三层限制？

**单层限制的问题**：
- ❌ 只能防止单一场景（突发或滥用）
- ❌ 无法灵活应对不同攻击模式
- ❌ 正常用户可能被过度限制

**三层限制的优势**：
- ✅ **分层防护**：每个层级应对不同威胁
- ✅ **灵活响应**：根据威胁级别调整限制强度
- ✅ **用户友好**：正常使用不受影响

### 具体防护场景

| 场景 | 触发层级 | 限制结果 |
|------|---------|---------|
| **脚本突发请求** | 短期（1分钟5次） | 立即阻止，提示"稍后再试" |
| **持续高频请求** | 中期（15分钟15次） | 15分钟阻止，提示"15分钟后再试" |
| **长期滥用** | 长期（1小时50次） | 1小时阻止，提示"1小时后再试" |

## 🔧 实现细节

### 技术栈

- **express-rate-limit**: Node.js 速率限制中间件
- **三层独立限制器**: 每层使用独立的计数器和时间窗口
- **响应头**: 标准 `RateLimit-*` 头信息
- **日志监控**: 自动记录触发限制的请求

### HTTP 响应头

当速率限制生效时，返回以下头信息：

```http
RateLimit-Limit: 5
RateLimit-Remaining: 0
RateLimit-Reset: 1708524300
Retry-After: 60
```

### 错误响应格式

```json
{
  "error": "请求过于频繁，请稍后再试。/ Too many requests. Please wait a moment.",
  "tier": "short-term",
  "retryAfter": "1 minute"
}
```

### 原配置 vs 新配置

| 指标 | 原配置 | 新配置（三层） | 改进 |
|------|--------|---------------|------|
| **配置方式** | 单层（15分钟100次） | 三层（短中长期） | ✅ 更灵活 |
| **突发防护** | ❌ 无 | ✅ 1分钟5次 | ✅ 新增 |
| **成本保护** | ⚠️ 一般 | ✅ 优秀 | ✅ 强化 |
| **用户体验** | ⚠️ 可能被滥用 | ✅ 平衡 | ✅ 优化 |
| **API 成本控制** | ⚠️ 弱 | ✅ 强 | ✅ 提升 |

### 与用户建议对比

| 限制 | 用户建议 | 实现方案 | 说明 |
|------|---------|---------|------|
| 1分钟 | 5次 | 5次 | ✅ 完全采纳 |
| 1小时 | 20次 | 50次 | ⚠️ 调整为更宽松 |

**为什么 1小时从20次调整为50次？**

- **20次过于严格**：正常用户日间使用可能受影响
- **事实核查特点**：每次请求5-15秒，50次/小时已足够严格
- **分层保护**：有短期和中期限制在前面，50次/小时不会成为瓶颈

**示例场景**：
```
正常用户：
- 上午核查10条新闻（30分钟内）
- 下午核查15条新闻（45分钟内）
- 晚上核查5条新闻（15分钟内）
总计：30次/天 ✅ 远低于50次/小时

恶意用户：
- 脚本1分钟内发送100次请求
→ 触发短期限制，立即阻止 ✅
```

## 🔒 安全增强建议

### 未来可扩展功能

#### 1. IP 黑名单

对于持续触发限制的恶意 IP，可以添加到黑名单：

```typescript
const BLACKLISTED_IPS = new Set(['1.2.3.4', '5.6.7.8']);

app.use((req, res, next) => {
  if (BLACKLISTED_IPS.has(req.ip)) {
    return res.status(403).json({
      error: 'IP banned / IP 已被封禁'
    });
  }
  next();
});
```

#### 2. 设备指纹

防止通过代理 IP 绕过限制：

```typescript
import { fingerprint } from 'express-device-fingerprint';

app.use(fingerprint());
// 使用设备ID + IP 组合作为限制标识
```

#### 3. CAPTCHA 验证

if (rateLimitExceeded && recaptchaFailed) {

}

## 🎯 最佳实践

### 对于正常用户

✅ **推荐使用方式**：
- 逐条核查（每条间隔1-2分钟）
- 批量核查（分批进行，每批5-10条）
- 避免脚本自动化

⚠️ **避免以下行为**：
- 连续快速点击提交按钮
- 使用浏览器自动刷新
- 并发多窗口同时核查

### 对于开发者

✅ **开发建议**：
- 前端添加防抖（debounce）按钮
- 显示剩余配额给用户
- 提供友好的错误提示
- 实现指数退避重试

⚠️ **注意事项**：
- 不要绕过客户端限制
- 尊重服务器速率限制
- 监控 API 使用成本
